## 1. Task Completion Notification

### Goal

Fire a VSCode information notification when a streaming response finishes, but only when the chat panel is not visible â€” so it acts as a tap on the shoulder when the user has switched back to editing code.

### Architecture

The change touches two files:

- **`src/chat/message-handler.ts`** â€” add an `onStreamEnd?: () => void` callback field; call it from the successful stream-completion path only (not from abort/cancel, not from error paths).
- **`src/extension.ts`** â€” after constructing `MessageHandler`, assign `messageHandler.onStreamEnd = () => { if (!chatProvider.webviewView?.visible) { vscode.window.showInformationMessage('Response ready'); } }`.

### When it fires

| Event | Fires? |
|---|---|
| Stream completes normally (`finish_reason: stop`) | âś… yes, if panel hidden |
| User cancels (abort) | âťŚ no |
| Network / API error | âťŚ no |
| Panel is visible | âťŚ no |

### No new settings

Notification only fires when panel is hidden, so it is never noisy. No `completionNotification` config key needed.

### Testing

One unit test in `message-handler.test.ts`: mock `onStreamEnd`, run a successful stream, assert it was called once. A second test: trigger abort path, assert `onStreamEnd` was not called.

## 2. Idiomatic Scroll-to-Bottom

### Goal

Replace 5 explicit `scrollToBottom()` call sites in `webview/src/App.tsx` with a single reactive `createEffect` that watches the `messages()` signal. Pure internal refactor â€” no user-visible behavior change.

### Change

```ts
// Remove all 5 manual scrollToBottom() calls.
// Add once, after messagesEndRef is assigned:
createEffect(() => {
  chatStore.messages(); // subscribe â€” fires on new messages and streaming chunks
  requestAnimationFrame(() => {
    messagesEndRef?.scrollIntoView({ behavior: 'smooth' });
  });
});
```

The `scrollToBottom` helper and `messagesEndRef` sentinel `<div>` are kept; only the call sites change.

### Why `requestAnimationFrame`

Ensures the DOM has rendered the new content before scrolling. Matches the existing imperative pattern.

### Testing

Existing scroll behavior is covered implicitly by the current test suite. No new tests needed â€” this is a refactor.
